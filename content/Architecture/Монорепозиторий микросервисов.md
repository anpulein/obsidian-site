## 🎯 Что такое монорепозиторий

> [!info] Определение **Монорепозиторий (Monorepo)** — это подход к организации кода, при котором несколько связанных проектов (микросервисов) хранятся в одном репозитории.

### 📊 Сравнение подходов

|Критерий|Полирепо|Монорепо|
|---|---|---|
|**Количество репозиториев**|Много (1 на сервис)|Один|
|**Управление зависимостями**|Сложное|Упрощенное|
|**Рефакторинг**|Затруднен|Простой|
|**CI/CD**|Распределенный|Централизованный|
|**Доступ к коду**|Ограниченный|Полный|

## ⚖️ Преимущества и недостатки

### ✅ Преимущества

> [!success] Основные плюсы
> 
> - **Единое место хранения** всего кода
> - **Атомарные коммиты** — изменения в нескольких сервисах в одном коммите
> - **Переиспользование кода** через общие библиотеки
> - **Упрощенный рефакторинг** кросс-сервисных изменений
> - **Единые стандарты** кодирования и инструменты

#### 🔄 Пример атомарного коммита

```bash
# Изменения в одном коммите затрагивают несколько сервисов
git commit -m "feat: add user authentication to all services
- services/auth: implement JWT validation
- services/user: add auth middleware  
- services/order: protect order endpoints
- shared/types: add User interface"
```

### ❌ Недостатки

> [!warning] Потенциальные проблемы
> 
> - **Размер репозитория** может стать огромным
> - **Права доступа** сложно гранулировать
> - **CI/CD время** увеличивается
> - **Конфликты слияния** при большой команде
> - **Инструменты** требуют специальной настройки

## 🔧 Инструменты и технологии

### 🛠️ Основные инструменты

#### 1. **Nx** - Система сборки и управления монорепо

> [!tip] Nx - лидер среди инструментов монорепо Nx предоставляет мощные возможности для управления большими кодовыми базами

```bash
# Установка Nx
npm install -g @nrwl/cli

# Создание workspace
npx create-nx-workspace@latest myorg

# Генерация нового сервиса
nx generate @nrwl/node:application user-service

# Запуск конкретного сервиса
nx serve user-service

# Запуск тестов для всех сервисов
nx run-many --target=test --all
```

#### 2. **Lerna** - Управление пакетами

```bash
# Установка зависимости для всех пакетов
lerna add lodash

# Установка зависимости для конкретного пакета
lerna add express --scope=user-service

# Публикация всех изменённых пакетов
lerna publish
```

#### 3. **Rush** - Корпоративный менеджер монорепо

```json
// rush.json
{
  "rushVersion": "5.82.0",
  "projects": [
    {
      "packageName": "auth-service",
      "projectFolder": "services/auth-service"
    },
    {
      "packageName": "user-service", 
      "projectFolder": "services/user-service"
    }
  ]
}
```

### 🛡️ Общий middleware

```typescript
// shared/middleware/auth.middleware.ts
import { Request, Response, NextFunction } from 'express';
import { JwtService } from '../utils/jwt.service';

export const authenticateToken = async (
  req: Request,
  res: Response,
  next: NextFunction
) => {
  try {
    const token = req.headers.authorization?.split(' ')[1];
    
    if (!token) {
      return res.status(401).json({ error: 'Access token required' });
    }

    const payload = await JwtService.verify(token);
    req.user = payload;
    next();
  } catch (error) {
    res.status(403).json({ error: 'Invalid token' });
  }
};
```

### 🎯 Использование в сервисах

```typescript
// services/order-service/src/controllers/order.controller.ts
import { User, Order } from '../../../shared/types';
import { authenticateToken } from '../../../shared/middleware';
import { OrderService } from '../services/order.service';

export class OrderController {
  @Get('/orders')
  @UseMiddleware(authenticateToken)
  async getOrders(req: Request, res: Response) {
    const userId = req.user.id;
    const orders = await OrderService.getOrdersByUserId(userId);
    res.json(orders);
  }

  @Post('/orders')
  @UseMiddleware(authenticateToken)
  async createOrder(req: Request, res: Response) {
    const orderData = req.body;
    const order = await OrderService.createOrder({
      ...orderData,
      userId: req.user.id
    });
    res.json(order);
  }
}
```

## 🚀 Развертывание

### 🐳 Docker конфигурация

#### Dockerfile для сервиса

```dockerfile
# services/user-service/Dockerfile
FROM node:18-alpine

WORKDIR /app

# Копируем root package.json для workspace
COPY package*.json ./
COPY nx.json ./

# Копируем shared библиотеки
COPY shared/ ./shared/

# Копируем конкретный сервис
COPY services/user-service/ ./services/user-service/

# Устанавливаем зависимости
RUN npm ci --only=production

# Собираем сервис
RUN npx nx build user-service

EXPOSE 3001

CMD ["node", "dist/services/user-service/main.js"]
```

### ⚙️ CI/CD Pipeline

```yaml
# .github/workflows/ci.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  changes:
    runs-on: ubuntu-latest
    outputs:
      auth: ${{ steps.changes.outputs.auth }}
      user: ${{ steps.changes.outputs.user }}
      order: ${{ steps.changes.outputs.order }}
    steps:
      - uses: actions/checkout@v3
      - uses: dorny/paths-filter@v2
        id: changes
        with:
          filters: |
            auth:
              - 'services/auth-service/**'
              - 'shared/**'
            user:
              - 'services/user-service/**'
              - 'shared/**'
            order:
              - 'services/order-service/**'
              - 'shared/**'

  test:
    needs: changes
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: [auth-service, user-service, order-service]
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run tests for ${{ matrix.service }}
        if: needs.changes.outputs[matrix.service] == 'true'
        run: npx nx test ${{ matrix.service }}
      
      - name: Build ${{ matrix.service }}
        if: needs.changes.outputs[matrix.service] == 'true'
        run: npx nx build ${{ matrix.service }}

  deploy:
    needs: [changes, test]
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v3
      
      - name: Deploy to production
        run: |
          # Деплой только измененных сервисов
          echo "Deploying changed services..."
```

### 📊 Мониторинг изменений

> [!note] Nx Affected Nx автоматически определяет, какие сервисы были затронуты изменениями

```bash
# Запуск тестов только для измененных сервисов
nx affected:test

# Сборка только измененных сервисов  
nx affected:build

# Деплой только измененных сервисов
nx affected --target=deploy
```

## 📈 Лучшие практики

### ✅ Do's (Рекомендации)

> [!success] Что нужно делать
> 
> 1. **Используйте единые стандарты** кодирования
> 2. **Автоматизируйте всё** — тесты, сборку, деплой
> 3. **Версионируйте общие библиотеки** семантически
> 4. **Документируйте API** каждого сервиса
> 5. **Настройте pre-commit hooks** для проверки качества

### ❌ Don'ts (Что избегать)

> [!danger] Чего не стоит делать
> 
> 1. **Не создавайте сильные связи** между сервисами
> 2. **Не храните секреты** в коде
> 3. **Не игнорируйте производительность** CI/CD
> 4. **Не смешивайте бизнес-логику** в shared библиотеках
> 5. **Не забывайте о безопасности** API
