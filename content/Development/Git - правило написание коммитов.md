# Правила написания коммитов по Conventional Commits 1.0.0

## 📑 Краткий обзор

> **Progressive summarization:** основные идеи выделены **жирным** шрифтом.

- **Conventional Commits** — спецификация для семантического, структурированного формата сообщений коммитов.    
- **Цели:** единый стиль, автоматизация создания CHANGELOG, улучшенная навигация по истории.
- **Ключевые элементы:** `type`, `scope`, `description`, `body`, `footer`.

---

## 🚀 Atomic Notes: основные понятия

- [[Type]] — категория изменения (`feat`, `fix`, `docs` и др.) 
- [[Scope]] — область проекта (модуль, компонент)
- [[Description]] — краткое описание цели коммита
- [[Body]] — подробное описание изменений
- [[Footer]] — метаданные (ссылки на задачи, BREAKING CHANGE)
- [[BREAKING CHANGE]] — явное указание на обратимо-не совместимые изменения

---

## 🔍 Детальное описание компонентов

### 1. Type

> **Обязательный** элемент. Указывает категорию изменений.

- **feat:** новая функциональность
- **fix:** исправление ошибки
- **docs:** изменения в документации
- **style:** форматирование, отступы, пробелы, точка с запятой и т.п.
- **refactor:** изменения кода без исправления ошибок или добавления функционала
- **perf:** улучшение производительности
- **test:** добавление или правка тестов
- **chore:** вспомогательные задачи (сборка, CI)
- **revert:** откат ранее сделанного коммита

### 2. Scope

> **Необязательный**, но рекомендованный. Уточняет, к какому модулю относится изменение.

```bash
# Пример: изменение в авторизации
feat(auth): добавить поддержку двухфакторной аутентификации
```

### 3. Description

> **Обязательный.** Краткое описание не более 50 символов, в императивном наклонении.

- Используйте **настоящие время**: `add`, `fix`, `update`, а не `added` или `fixed`.

### 4. Body

> **Необязательный.** Детальное описание изменений.

- Разбивайте на абзацы по смыслу.
- Если есть мотивация или ссылку на задачу, укажите её здесь.

```text
Развернута логика в методе `validateUser()`, чтобы учитывать новые правила безопасности:
- добавлена проверка пароля по регулярному выражению
- обновлены юнит-тесты для покрытия новых кейсов
```

### 5. Footer

> **Необязательный.** Используется для ссылок на задачи или маркеров BREAKING CHANGE.

- `Closes #123` — автоматически закроет issue №123.
- `BREAKING CHANGE: описание` — указывает на несовместимые изменения.

## 📝 Таблица типов коммитов

|Тип|Описание|Пример|
|---|---|---|
|`feat`|Новая функциональность|`feat(payment): добавить поддержку Apple Pay`|
|`fix`|Исправление ошибки|`fix(api): обработать ошибку 500`|
|`docs`|Изменения в документации|`docs(readme): обновить инструкцию по установке`|
|`style`|Форматирование кода|`style: привести отступы к единому виду`|
|`refactor`|Изменение структуры кода без новой функциональности|`refactor(user): вынести валидацию пароля`|
|`perf`|Улучшение производительности|`perf: оптимизировать загрузку изображений`|
|`test`|Добавление или правка тестов|`test: добавить тесты для оплаты через API`|
|`chore`|Вспомогательные задачи (сборка, CI, deps)|`chore: обновить зависимости`|
|`revert`|Откат ранее сделанного коммита|`revert: feat(auth): добавить 2FA`|

## 💡 Советы и лучшие практики

- **Императивное наклонение:** описывайте действие, например `add`, `remove`, `update`.
- **Краткость:** держите описание ≤50 символов, тело — ≤72 символов на строку.
- **Единообразие:** используйте один стиль во всех коммитах проекта.
- **Автогенерация CHANGELOG:** многие инструменты (например, [semantic-release]) поддерживают Conventional Commits.
- **Проверка через хуки:** настройте [`commitlint`](https://github.com/conventional-changelog/commitlint) для автоматического контроля формата.

> **Цитата:**  
> “Conventional Commits were inspired by the Angular commit message conventions.”

## 🛠 Примеры коммитов

```bash
feat(ui): добавить компонент карточки товара

Разработан новый компонент `ProductCard`:
- отображение изображения и цены
- кнопка "Добавить в корзину"

Closes #45
```

```bash
fix(api): корректная обработка ошибок 404

Раньше при отсутствии ресурса возвращался статус 200.
Теперь возвращается 404 и сообщение об ошибке.
```

```bash
chore: обновить зависимости

- Обновлен `webpack` до v5.XX
- Исправлены конфликты в `package.json`
```
